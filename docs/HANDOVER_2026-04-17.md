## 0. プロジェクト一言サマリ

**GALOSH = Generalized Anscombe LOcal SHrinkage** — ノイズパラメータ事前情報**ゼロ**、学習データ**ゼロ**、GPU **なし**でも動く **fully-blind local denoiser**。GAT (Generalized Anscombe Transformation) でノイズを i.i.d. Gaussian (σ=1) に正規化してから、**local WHT shrinkage だけ**で block-matching に匹敵する品質を出す。

**Core thesis**: BM3D の block matching は「ノイズが非定常/未知」を補うためにある。GAT + σ 正規化で分散が一定なら、**block matching は不要**。

**現時点の最強スポット**: SIDD Medium 80 枚で **blind non-learning 系 No.1**、さらに **sensor 非特化 DL (DnCNN-B / DRUNet / FFDNet / SwinIR / Restormer / SCUNet) を効率で圧倒**、かつ SIDD 特化学習の NAFNet と同じ土俵で perceptual 品質を戦える。

## 1. コンセプト・設計思想

### 1.1 原点の動機 (TYPE-B design intent)

**ゴール**: WB・demosaic 前に**最高品質の RAW denoise** を実現する。

**なぜ pre-WB/demosaic denoising か**:
1. RB にノイズが乗った状態で WB ゲインをかけると暗部でマゼンタ浮きが発生する
2. demosaic 時にノイズがあると補間がノイズに引っ張られて誤つなぎ + 偽色発生

**Core approach**:
- 輝度 (L) でディテール保証、色差 (C) でノイズ抑制 (L/C 分離デノイズ一般則)
- これを **RGGB RAW に直接適用** (半解像度 L/C = 2×2 Walsh-Hadamard decomposition)
- GAT による **fully blind** (ノイズプロファイル不要)
- GAT が優秀なので RGB や YUV への展開も自然にできる

**重要**: これは BM3D のフィルタリング方式の問題ではない。**処理プロセス全体の設計**が肝。初期には「GAT 前に WHT」「最初に WHT」等も検討した。手段 (Wiener の種類) に固執せず、目的 (WB 前最高品質 RAW denoise) から逆算して設計する。

### 1.2 何を重視する (user priorities)

| 項目 | 方針 |
|---|---|
| **メイン評価指標** | **LPIPS (perceptual)** ≥ PSNR、表示順は LPIPS / SSIM / Luma PSNR / Chroma PSNR |
| **Blindness** | ノイズ σ や α の事前情報不要、カメラプロファイル不要 |
| **効率性** | zero training data, zero parameters, **no GPU required** が差別化の核 |
| **データ保存** | 全手法の output PNG + GT + Noisy を必ず保存 (視覚比較データとして保全) |
| **開発原則** | 勝手にコード書かない、分析 → 許可 → 実装 |
| **コメント** | 理論アップデートは EN + JP 両言語でコード内に、paper-ready に自己完結 |

### 1.3 2 つの "最強" 主張

PR の主戦略は下記 2 軸:

1. **"Blind non-learning 最強"** — BM3D-CFA / CBM3D / NLM / Guided / BM3D 系全てに勝つ
2. **"Sensor 非特化 DL 含めて最強"** — DnCNN-B / DRUNet / FFDNet / SwinIR / Restormer / SCUNet の AWGN/blind 学習済モデルに勝つ
3. SIDD 特化学習の NAFNet (PSNR=42.23) は別枠 (「データがあれば負けるが、データなしで勝つ」ポジション)

## 2. アルゴリズム詳解

### 2.1 GALOSH_F (CPU 版、現行 best: stride=2)

**アーキテクチャ** (`standalone/rawdenoise_v6.c` + `#define GALOSH_F`):
```
RAW Bayer (RGGB)
  ↓
GAT (Poisson-Gauss VST: 2√(αx+σ²))        ← ノイズ分散一定化
  ↓
Half-res 2×2 WHT L/C decomposition       ← L(DC) + C1/C2/C3(AC)
  ↓
Sigma normalization (per-band)
  ↓
┌─ Half-res L: BayesShrink Pass1 + Wiener Pass2 (stride=2)
├─ Half-res C1/C2/C3: 同上 (stride=2)
└─ Full-res L: compute_L_fullres + Wiener Pass2 (stride=2)
  ↓
Block-aligned L reconstruction (inverse 2×2 WHT)
  ↓
Inv-GAT (Makitalo-Foi exact unbiased inverse)
  ↓
RAW Bayer (denoised)
```

**キー性質**:
- No block matching, no search window, no sorting → **全ブロック独立 → fully GPU parallelizable**
- stride=2 (75% overlap) が stride=4 比 +0.25 dB PSNR、LPIPS/DISTS も改善、速度は 1.5× のみ
- σ_L / σ_C を独立に制御できる (ユーザーの luma grain 好み調整可能な UX)

**Ablation (2026-04-12)**:
- F_HALF only: 改善の 80%
- F_FULL only: 改善の 20%
- F (both): 加算的に最大効果

**Kodak 結果 (σ_L=1.0 / σ_C=1.0)**:
- ISO 400: PSNR=34.07, LPIPS=0.0841, DISTS=0.0827, 0.12s/img — **oracle BM3D-CFA を LPIPS で超える**
- ISO 800: PSNR=31.66, LPIPS=0.1229, DISTS=0.1203
- ISO 3200: PSNR=25.78, LPIPS=0.3192, DISTS=0.2358

**重要な学習**: Full-res raw GALOSH の CFA protection (案A、WHT indices {0,1,8,9} を shrinkage から保護) は**暗部マゼンタ色シフト修正に不十分**と確認済 (2026-04-12)。6.25% (4/64) の保護では不足、AC 係数の色情報も過度に shrink される。**唯一 proven な color-shift-free 構造が案B (GALOSH_F / GALOSH_FULLRES_L)** で、これが現行本命。

### 2.2 GPU ファミリー (本セッションでフォーカス)

**3 ID 共存、ライバルではなく入力 domain で使い分け**:

| ID | 入力 | 実装 | 役割 |
|---|---|---|---|
| `galosh_raw_gpu` | RAW Bayer (pre-demosaic) | `rawdenoise_gpu.c` + `galosh_fused.cl` (OpenCL, ~2336 行) | カメラ内 RAW パイプ、CFA native denoising |
| `galosh_yuv_gpu` (new) | sRGB → linear YCbCr | `rawdenoise_gpu.c` + `galosh_yuv_gat.cl` (OpenCL 新規 696 行) | **linear Y-GAT + chroma VST**、改良版 |
| `galosh_yuv_gpu_old` | sRGB plain YUV (no GAT, plain Wiener) | `rawdenoise_gpu.c` 旧モード | **ablation baseline** |

#### 2.2.1 galosh_raw_gpu

**特徴**:
- Foi+Alenius 2008 blind noise estimation を GPU kernel で実装 (dark histogram + Laplacian histogram + 最小二乗回帰)
- 全 21 OpenCL kernels、CPU ↔ GPU 転送は入口 1 回 + 出口 1 回のみ
- ISP 物語: 「ワンチップ GPU/ISP で完結する proof-of-concept」 — CPU は信号処理に触らない
- PARAMS_SIZE 32 float (α, σ², 各種 LUT 係数)

#### 2.2.2 galosh_yuv_gpu (新版) — 本セッションの改良ターゲット

**コア洞察 (ユーザー発案の物理的観察)**:
> 「Cb/Cr のノイズは突き詰めれば R/G/B の shot+read noise の線形結合。ただし分散は **Cb の値ではなく、その画素の明るさ Y** で決まる。よって Y の blind estimation 1 回で Y/Cb/Cr 全ての per-pixel 分散マップが出る。」

**理論** (`galosh_yuv_gat.cl` 冒頭コメント、paper 引用級に整備済):

| plane | ノイズ分布 | 分散 | VST |
|---|---|---|---|
| linear Y = 0.2126R + 0.7152G + 0.0722B | Poisson-Gauss | `Var[Y] ≈ α_Y·Y + σ²_Y` | **非線形 GAT** `2√(α_Y·Y+σ²_Y)` |
| linear Cb = (B-Y)/1.8556 | Gaussian | `Var[Cb] ≈ α_Cb·Y + σ²_Cb` (**Y 駆動**) | **線形 VST** `Cb/√(α_Cb·Y+σ²_Cb)` |
| linear Cr = (R-Y)/1.5748 | Gaussian | `Var[Cr] ≈ α_Cr·Y + σ²_Cr` | 同上 |

**Closed-form パラメータ** (BT.709 + iid shot noise 仮定):
```
α_Cb ≈ 0.375·α_Y    σ²_Cb ≈ 0.375·σ²_Y
α_Cr ≈ 0.605·α_Y    σ²_Cr ≈ 0.605·σ²_Y
```

**パイプライン**:
```
noisy_srgb (H,W,3) float32
  ↓ sRGB EOTF⁻¹ (shot noise 則を復活)
linear_rgb
  ↓ BT.709 linear YCbCr 変換
Y_lin / Cb_lin / Cr_lin
  ↓ Foi+Alenius blind estimate on Y → (α_Y, σ²_Y) → 解析式で (α_Cb, σ²_Cb), (α_Cr, σ²_Cr)
  │
  ├─ Y 経路 (非線形):
  │     Y_stab = 2√(α_Y·Y+σ²_Y) → LOSH → Makitalo-Foi inverse
  │
  ├─ Cb 経路 (線形):
  │     Cb_stab = Cb_lin / √(α_Cb·Y+σ²_Cb) → LOSH → × √(...)
  │
  ├─ Cr 経路: 同上
  │
  └─ bivariate Wiener (Cb, Cr joint 2×2 共分散)
  ↓
YCbCr → linear RGB → sRGB OETF → out
```

**VST 非対称性の意図**:
| 経路 | ノイズ | 前方 VST | 逆 VST | bias 補正 |
|---|---|---|---|---|
| Y | Poisson-Gauss | `2√(αY+σ²)` 非線形 | Makitalo-Foi exact | **必要** |
| Cb/Cr | Gaussian | `÷√(αY+σ²)` 線形 | `×√(αY+σ²)` 線形 | **不要** (線形演算で exact) |

両経路とも「前方 VST → LOSH → 逆 VST」の同一サンドイッチ構造。ノイズ分布の違いで VST の形 (非線形/線形) が分岐するだけで、枠組みは一貫。

**観察**:

- `galosh_yuv_gpu` (new) は現状 PSNR で RAW より 4 dB 劣る — GAT + chroma VST 理論は正しいが実装の詰めが甘い

### 3.1 galosh_yuv_gpu 改良したい意図 (最重要、現ターゲット)

**現状 PSNR=28.50 / LPIPS=0.667 は期待値を下回る**。以下 7 項目の最適化で

- **PSNR: +0.5 〜 +1.0 dB**
- **LPIPS: -0.05 〜 -0.08**
- **NIQE: -0.5 〜 -1.0**
- **速度: 2〜3× 高速** (0.85s → 0.3-0.4s/img 目標)

を狙う:

#### 品質最適化 (3 項目)

1. **Multi-scale LOSH (Laplacian pyramid 3-level)** — 低周波色ムラ vs 高周波テクスチャ分離 (+0.3〜0.8 dB / LPIPS -0.02〜-0.05)
2. **Bivariate joint (Cb, Cr) Wiener** — 2×2 noise + 2×2 signal 共分散で joint shrinkage、色線 (blue-yellow / red-cyan) coherence 向上 (LPIPS -0.02〜-0.05)
3. **経験的 chroma blind estimation** — 解析式でなく Cb 平面を直接 Foi+Alenius、colorful scene での gray-scene 近似崩壊回避 (+0.1〜0.3 dB)

#### 速度最適化 (5 項目、GPU + LDS 32KB 制約下)

4. **FP16 compute path + FP32 Makitalo LUT (LDS 常駐)** — GAT/LOSH/inverse は fp16、Makitalo 補正テーブル (~2KB) を LDS shared (1.5〜2× 速)
5. **Fused sRGB→linear→YCbCr 単一 kernel** — register 内連結、global memory round-trip 2→1 (全体 -5〜-10%)
6. **Tile+halo LOSH kernel with LDS** — Y kernel 単独で 80×80 tile + halo 4 = 15.5 KB (32KB の半分未満)、chroma kernel 別で ~18 KB (2〜4× 速)
7. **Pre-compute `α·Y + σ²` scratch plane** — Y-GAT / Cb scale / Cr scale 3 経路で共有 (~5%)
8. **Separable LOSH 1D×2** — 2D N×N → 2×N flops (N=7 で 3.5× 軽量)

#### 反復チューニング (限界まで詰めるループ)

**チューニングパラメータ**:
- LOSH patch size: {5, 7, 9}
- LOSH λ (frequency-band 別): {0.5, 1.0, 1.5, 2.0} × σ_stab
- Pyramid levels: {2, 3, 4}
- FP16 precision: Makitalo LUT 補間 {256, 512, 1024} entries
- Tile size: {48, 64, 80, 96}
- Bivariate Wiener regularization ε: {1e-4, 1e-3, 1e-2}

**評価フロー**:
1. Phase F 基本実装 → 10 枚 subset → PSNR/LPIPS/NIQE 確認
2. ハイパー掃引 (grid search ~30-50 runs、1 run = 10 枚 × 1s = 10s、合計 ~10 分)
3. 最良設定で 80 枚本番 (~2 分)
4. 必要なら residual analysis → 弱点特定 → patch / λ 再調整 → 再 bench

**収束判定**: LPIPS が 0.001 未満の変化で 3 ラウンド連続 → 収束 (または 10 ラウンド打ち切り)

## 4. 論文戦略

### 4.1 Paper selling points (既存の v2 multichannel GAT+BM3D 時点の PR)

1. Quality on par with BM3D-CFA (**LPIPS better**, PSNR slightly lower) at **13% less compute**
2. **75× more efficient than DnCNN-B** for nearly identical quality
3. **233× more efficient than DRUNet** (which wins on pure quality)
4. Tops efficiency-weighted composite scores (Q/log2C, Q/C^0.2)
5. **Zero training data, zero parameters, no GPU required**

→ どのベンチマーク報告 / README / paper draft にも **効率性優位** を必ず含める。FLOPs 比較を PSNR/SSIM/LPIPS と並置。

### 4.2 Paper 投稿戦略

- **arXiv preprint** → **IEEE submission** 予定 (image processing 系ジャーナル想定、想定誌: IEEE TIP / Signal Processing: Image Communication / JEI)
- 想定タイトル (draft): "GALOSH: Generalized Anscombe LOcal SHrinkage — A Block-Matching-Free Blind Denoiser with Real-Time 4K Potential"

### 4.3 主要主張 (paper で立証すべき命題)

1. **GAT + σ 正規化により block matching が不要である** (BM3D の non-local 仮定を崩す)
2. **GPU 時代の最適形**: 全ブロック独立 → work group 並列、block matching の不規則メモリアクセス無し
3. **L/C 分離 + GAT の独立性** により **fully blind** 運用可能
4. **sRGB 入力への拡張が自然** に可能 (linear Y-GAT + Y-driven chroma VST、1 回の blind estimation で 3 plane)
5. **Kodak oracle BM3D-CFA を LPIPS で超える最初の fully-blind local denoiser**

### 4.4 Ablation 表 (paper で必須)

- GALOSH_F vs F_HALF vs F_FULL (半/全 res 貢献分離)
- GALOSH_RAW_GPU vs GALOSH_YUV_GPU (nogat) vs GALOSH_YUV_GPU (new, linear Y-GAT + chroma VST) — **VST 有無の効果を純粋分離**
- σ_L / σ_C 独立制御 vs fixed

### 4.5 比較ベースライン (SIDD Medium 16 手法構成)

**Rule-based**: GALOSH vs BM3D-CFA / CBM3D / NLM / Guided / BM3D-sRGB
**DL sensor 非特化**: FFDNet / SwinIR / Restormer / SCUNet / DnCNN-B / DRUNet
**DL sensor 特化** (上限参考): NAFNet

**Claim verification**:
- GALOSH ≥ BM3D-CFA on (PSNR/LPIPS/NIQE) → **"Blind non-learning 最強"** 確定
- GALOSH ≥ min(FFDNet/SwinIR/SCUNet/Restormer) → **"Sensor 非特化 DL 含めて最強"** 確定

## 5. 特許戦略 (推定、要ユーザー確認)

memory には明示的な特許計画なし。ただし **特許性のある novel contributions** と思われるもの:

### 5.1 潜在的 claim 候補

1. **"linear-domain Y-GAT + Y-driven chroma VST"** アーキテクチャ
   - Cb/Cr の分散が **Cb 自身ではなく Y で決まる** という洞察 (BT.709 + iid shot noise)
   - α_Cb/α_Cr を α_Y から **closed-form で導出** し、1 回の blind estimation で 3 plane カバー
   - 非線形 VST (Y) と線形 VST (Cb/Cr) の**非対称同時処理**
2. **"Bivariate (Cb, Cr) joint Wiener with Y-driven noise covariance"**
   - Y 駆動の 2×2 noise 共分散行列 + local 2×2 signal 共分散による joint shrinkage
3. **"Fully blind RAW L/C BM3D via GAT"** (TYPE-B design)
   - Pre-WB/demosaic RAW に対する半解像度 L/C 分離 + GAT fully blind
   - WB 前・demosaic 前に適用することによる偽色・暗部マゼンタ防止
4. **"Stride-2 block-aligned full-res L reconstruction"** (GALOSH_F)
   - CFA protection 不要の color-shift-free 構造

### 5.2 出願方針 (要ユーザー検討事項)

- **優先権主張**: arXiv preprint 公開前に出願すべき
- **対象**: 日本優先 + PCT 出願検討 (ISP/camera メーカー向け)
- **クレーム構造**: 独立クレーム = 「GAT による blind VST → local shrinkage → inverse VST」の sandwich、従属クレームで L/C 分離 / Y 駆動 chroma VST / stride-2 / GPU 実装を個別化

## 7. ユーザー嗜好 + 作業原則 (重要、毎回遵守)

### 7.1 コミュニケーション
- **チャットは必ず日本語** (コード/コメントは英語 OK)
- 評価指標の表示順: **LPIPS / SSIM / Luma PSNR / Chroma PSNR**
- ベンチは自動化 + HTML viewer で視覚比較

### 7.2 開発プロセス
- **勝手にコード書かない** — 分析 → 承認 → 実装。承認語彙: 「やって」「OK」「お願い」「GO」
- 「これからは全手法の結果画像で残しておいてー できればGTも同じ形式で置いておいて」— ベンチごとに **全手法の PNG + GT + Noisy** 保存

### 7.3 コメント基準
- 非自明な変更 (特に理論/アルゴ更新) は常にコード内に詳細コメント
- **バイリンガル EN + JP**、プロジェクト標準
- **理論 rationale** ("what" ではなく "why")
- **Self-contained** — 内部ステージ名 ("Stage 2 B2", "Option α", "策B+C" 等) を in-place で使わず、section header の導出を参照
- **Pipeline overview** の section header は常に現状反映 (新しい理論ハイライト含む)
- **後で疑問視されそうな設計判断の rationale** (unified σ vs per-ch, self-consistent vs fixed scale 等)

理由:
1. paper 執筆時にコメントが draft テキストになる
2. 再読時の理論整合性確認が楽
3. デバッグ時に前提条件が explicit
